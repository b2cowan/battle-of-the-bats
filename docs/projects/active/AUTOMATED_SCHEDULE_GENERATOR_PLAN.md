# No-Cost Automated Schedule Generator Plan

## Goal

Upgrade the tournament round-robin generator from a simple first-fit fixture placer into an automated scheduling assistant that can score, compare, and improve schedules using only code we own and data already in FieldLogicHQ.

This project must not introduce paid map, geocoding, route-matrix, drive-time, optimization, or SaaS dependencies. All scheduling logic should run in our application code using existing tournament, division, team, venue, facility, and game data.

## Product Scope

The generator should help an organizer answer three questions before saving a schedule:

1. Did every team get the expected number of games?
2. Is the schedule fair enough across rest, days, venues, and time slots?
3. What tradeoffs did the generator make, and can the organizer improve them before publishing?

The saved schedule page should show the same analytics after submission so an organizer can keep auditing the schedule as manual edits happen.

## Explicit No-Cost Guardrails

In scope:

- Internal schedule metrics and fairness scoring.
- Internal round-robin pairing and partial round-robin balancing.
- Internal scored slot assignment heuristic.
- Existing venue/facility data.
- Organizer-entered constraints and preferences.
- Optional manual venue-to-venue travel buffer values entered by the organizer.
- Existing Next.js, React, TypeScript, Supabase, and test tooling already in the repo.

Out of scope:

- Google Maps, Routes, Distance Matrix, Geocoding, Places, or Map display APIs.
- Mapbox, HERE, TomTom, OpenRouteService hosted APIs, or other paid/hosted routing APIs.
- Paid optimization engines or commercial solver licenses.
- Drive-time estimates generated by an external service.
- Permanent storage of third-party geocodes or route results.
- Self-hosted routing infrastructure in this phase, because it avoids API fees but adds operational cost and maintenance.

## Current Codebase Findings

- The current generator lives in `app/[orgSlug]/admin/tournaments/schedule/Generator.tsx`.
- It supports team-based and slot-based generation, division selection, games per team/slot, date windows, duration, turnover, and venue selection.
- It builds parent-venue time slots, creates round-robin matchups within each pool, and greedily assigns the first slot that does not double-book a team at the exact same time.
- Existing conflict utilities live in `lib/schedule-conflict.ts` and already support venue/facility overlap and buffer warnings for saved/manual schedule editing.
- The schedule page already receives `games`, `teams`, `divisions`, `venues`, and `currentTournament`, which is enough to compute saved-schedule metrics without a new API.
- The current generator writes games through `/api/admin/games` bulk-save and clears division games before insert.
- Venue hierarchy exists: `Venue` has child `VenueFacility[]`, and games support `venueFacilityId`.

## Architecture

### New Shared Libraries

1. `lib/schedule-metrics.ts`
   - Pure TypeScript utilities for generated and saved schedules.
   - Computes health score, issue list, team metrics, day metrics, venue/facility metrics, fairness ranges, and summary cards.
   - Accepts both persisted `Game[]` and draft games without IDs.
   - Handles team-based and slot-based placeholder schedules.

2. `lib/schedule-generator.ts`
   - Pure TypeScript generator engine.
   - Builds matchups, candidate slots, hard constraint checks, soft scoring, and draft options.
   - Does not import React.
   - Does not perform fetches.
   - Designed for unit testing.

3. Optional follow-up: `lib/schedule-constraints.ts`
   - If `schedule-generator.ts` grows too large, split reusable hard/soft constraint logic into this file.

### Data Model

V1 avoids database migrations where possible, except for durable schedule concepts that need to survive across sessions: temporary-facility lanes and manual generator locks.

- Draft metrics are computed in memory before commit.
- Saved schedule metrics are computed from existing `games`.
- Generator form settings live in component state.
- Tournament-level defaults can reuse existing `tournaments.settings` if we decide to persist generator defaults later.
- `schedule_facility_lanes` stores temporary schedule resources per tournament/division.
- `games.schedule_facility_lane_id` links generated games to a temporary lane until it is resolved.
- Resolving a lane updates the lane record plus all linked games' `diamond_id`, `venue_facility_id`, and display `location`.
- `games.generator_locked` lets an organizer keep a scheduled round-robin game fixed during Build from current regeneration.

Potential later no-cost persistence:

- Add `scheduleGeneratorDefaults` to `tournaments.settings`.
- Add manual travel buffer map to `tournaments.settings`, for example:
  - `schedule_manual_travel_buffers: Record<string, number>`
  - key format: `venueIdA::venueIdB`

No migration is required for new `tournaments.settings` keys, but API whitelisting must be updated if saved through `patch-settings`.

## Metrics Model

### Schedule Health Score

Health score: 0 to 100.

Suggested breakdown:

- Game count balance: 20 points
- Rest fairness: 20 points
- Day load fairness: 15 points
- Venue/facility movement: 15 points
- Time-slot fairness: 15 points
- Conflict safety: 15 points

The exact weights should be constants in `schedule-metrics.ts` so they are easy to tune.

### Draft and Saved Metrics

Compute:

- Total games.
- Expected games.
- Teams at target game count.
- Teams under target.
- Teams over target.
- Min, max, and average games per team.
- Games per team by day.
- Max games per team per day.
- Back-to-back count.
- Minimum rest per team.
- Long idle gaps per team.
- Venue changes per team.
- Facility changes per team.
- Cross-venue changes per team.
- Same-venue facility changes per team.
- Manual travel-buffer violations, if organizer-entered buffers exist.
- Early, midday, late game distribution per team.
- First-game and last-game distribution per team.
- Venue/facility utilization by day.
- Conflict count from existing conflict logic.
- Pool-by-pool schedule completeness.
- Rematch count, if the requested game count requires or permits repeats.

### Issue Severity

Metrics should produce a ranked issue list:

- `error`: must fix before commit or publish.
- `warning`: allowed but should be reviewed.
- `info`: useful context.

Examples:

- Error: team has fewer games than requested.
- Error: facility overlap.
- Warning: team has 3 games in one day.
- Warning: team has 2 back-to-back games.
- Warning: team changes venues with less than the manual travel buffer.
- Info: one venue is unused.
- Info: late games are concentrated on two teams.

## Generator Parameters

### Basic Controls

- Division.
- Generation mode: team-based or slot-based.
- Games per team or slot.
- Date windows.
- Facility selection, not only parent venue selection. **Implemented 2026-06-01.**
- Game duration.
- Turnover time.

### New Constraint Controls

Hard constraints:

- Minimum rest between games per team.
- Max games per team per day.
- Prevent facility overlap.
- Prevent parent-venue overlap when no facility is selected.
- Prevent team double-booking.
- Respect selected dates and facility availability.
- Respect locked existing games when partial regeneration is active.

Soft preferences:

- Avoid back-to-back games.
- Prefer same venue for a team's same-day games.
- Prefer same facility when practical.
- Minimize venue changes.
- Minimize long idle gaps.
- Balance early and late games.
- Balance facility usage.
- Keep pool games grouped by venue or time block where practical.
- Prefer compact schedule length.

### Presets

Add preset buttons that set weights and constraints:

- Balanced
- Compact
- Rest-friendly
- Venue-friendly
- Younger divisions earlier

Presets should be simple configuration objects, not separate algorithms.

## Generation Algorithm

### Phase 1 Algorithm: Scored Heuristic

Do not add a solver dependency in V1. Build an internal heuristic first.

1. Build matchups.
   - Full round robin when possible.
   - Balanced partial round robin when `gamesPerTeam < poolSize - 1`.
   - Explain impossible exact counts for odd team counts.
   - Keep pool boundaries unless a future crossover option is selected.

2. Build candidate slots.
   - Use selected date windows.
   - Use selected facilities.
   - Fall back to venue-level slots only when a venue has no facilities.
   - Sort by date/time, then venue/facility display order.

3. Assign games.
   - For each matchup, find candidate slots that pass hard constraints.
   - Score each candidate using soft preference penalties.
   - Choose the lowest-penalty slot.
   - Track team state as games are assigned.

4. Multi-pass improvement.
   - Generate several drafts using deterministic seeded shuffle variations.
   - Score each complete draft with `schedule-metrics.ts`.
   - Keep the top 3 options.

5. Local repair pass.
   - Identify worst issues.
   - Try swaps between games that reduce total penalty without violating hard constraints.
   - Limit attempts to keep UI responsive.

### Determinism

The first generated draft should be reproducible for the same inputs.

- Use a deterministic seeded pseudo-random function.
- Include a "Generate another set" action that changes the seed.

### Performance Guardrails

- Keep all generation client-side for V1.
- Add a maximum attempt count.
- Add a time budget, for example 500 ms to 1500 ms per generate action.
- If the search limit is hit, return the best draft plus a clear warning.

## UX Plan

### Generator Form

Replace the single long form with a compact sections model:

1. Basics
   - Mode, division, games per team/slot.

2. Availability
   - Dates and time windows.
   - Facilities grouped under venues.

3. Rules

- Health score.
- Top issues.
- Summary metrics.
- Per-team fairness table.

### Saved Schedule Analytics

On the schedule page, add a compact Schedule Health surface:

- Visible for Round Robin view.
- Computes from saved games.
- Can be collapsed.

Implemented UX:

- "Regenerate schedule" can preserve existing games.
- Organizer can choose:
  - Replace all games in division.
  - Keep completed/submitted games.
  - Keep manually locked scheduled games.
  - Replace only unlocked scheduled round-robin games.

## Success Criteria

- Organizers can see objective quality metrics before committing generated games.
- The schedule page exposes the same health view after games are saved.
- Generator can avoid basic unfairness without paid services:
  - fewer back-to-backs
  - better rest spacing
  - fewer venue/facility changes
  - balanced early/late assignments
  - max games/day respected
- The generator explains impossible or tradeoff-heavy schedules instead of silently producing poor output.
- Facility-level schedule generation persists `venueFacilityId`.
- No paid API keys, external route services, or commercial solver licenses are required.

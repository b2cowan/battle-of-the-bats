## Invited staff who don't use the email link get dropped into "Create Your Organization" — 2026-07-06
- **Symptom:** A person invited as staff/admin to a tournament opens their invite email, but instead of clicking the "Accept Invitation" button they reach the site through a generic "Get started / Create account" entry (including the login page's "New organization? Create account" link). That front door defaults to the **owner** signup — "Create Your Organization" — so instead of being carried into the tournament they were invited to, they're prompted to start their own org. Their Members-page row stays **PENDING** even though they've created a working account (Last Sign In stamped), and they may create a **stray org** they don't need. Real customer hit: `bryan.doucette@live.com` invited to Milton Softball Organization, got the org-creation prompt, invite never completed.
- **Why no self-serve fix (for PREVENTION):** The *recovery* is self-serve — the Members "Resend" button sends a magic link straight to the accept-invite screen, and platform-admin Customer Users shows any stray org — so the individual ticket can be closed without engineering. What has **no lever** is prevention: the generic signup does **not** check whether a pending invite already exists for the entered email, offers no "Accept your invitation instead" branch, and nothing stops an invitee from creating a stray org. The correct paths exist (email link + the "I was invited" account-only tile) but a user who takes the wrong door gets no redirect back onto the rails. This will keep recurring per invite wave.
- **Suggested surface:** At signup (and ideally on the `/start` chooser), detect a pending invite matching the entered email and surface an "You've been invited to {org} — Accept invitation" branch instead of / above org creation; optionally soft-block or warn an invited email from creating a brand-new org. This is a signup/onboarding **flow** fix, not a platform-admin tool.
- **Workaround used:** Confirm no stray org (platform-admin Customer Users / read-only query), then click **Resend** on the member's row — the magic link lands them on accept-invite and into the tournament; tell them to use the email button, not "Create account."
- **Effort:** S (detect pending invite + "Accept instead" CTA on signup) / M (also cover `/start` + invited-email org-create guard)
- **Related:** [[reference_invite_auth_flow]] / invite reconciliation (mig 128) / signup–org decoupling ("I was invited" path) / F-series support-seam work

## Removing an org admin who is ALSO a free coach silently deletes their whole free Coaches Portal — date TBD
- **Symptom:** An org owner/admin removes a staff member from the org Members page. If that person's ONLY org membership is this org, but they separately run a **free (Basic) Coaches Portal** — their own team(s) with roster, players, fees, announcements, tournament history — the Remove action hard-deletes their entire account AND deletes every free coach team they solely own. The free portal and all its data are gone, and the person gets **no notification at all** (Remove sends no email; only Suspend does). The removal-confirmation warning shows the generic "permanently delete their account" copy with **no mention** that a coaching portal will be destroyed.
- **Why no self-serve fix:** The "keep their account, remove membership only" safeguard (J4-036) only triggers when the person belongs to **another organization**. A free Coaches Portal is **org-less** (Basic coach teams aren't organizations), so it doesn't count — the person reads as "sole membership" and falls into the hard-delete path. The pre-removal impact warning only counts **paid** rep-team coaching assignments; it is **blind to free Basic coach teams**, so the admin gets zero signal that they're about to wipe out someone's coaching portal. There is no platform-admin surface that shows "this user also has a free coaching portal" or that can undo the deletion afterward.
- **Suggested surface:** (1) Broaden the removal safeguard so ANY off-org presence — including a free Basic Coaches Portal — forces the **membership-only** path (preserve the account; just drop this org's access), mirroring the J4-036 intent. (2) Make the pre-removal impact check also count Basic coach teams and surface an amber warning ("This person runs N free coaching team(s) — removing them will permanently delete that portal"). (3) Consider sending the removed person a notice, matching the Suspend behavior.
- **Workaround used:** Operator must manually check whether the admin also coaches a free team **before** clicking Remove (no UI shows this) and avoid removing them, or accept the data loss. No clean self-serve path today.
- **Effort:** S (add the free-team count to the warning) / M (broaden the membership-only safeguard + notice)
- **Related:** [[reference_coach_portal_arch_decision]] / [[project_coaches_portal_architecture]] / J4-036 (membership-only removal) / F-series support-seam work

## No prod migration-drift visibility — code shipped ahead of its DB migration 500s in prod — 2026-06-22
- **Symptom:** Publishing a division schedule in prod returns 500 with a useless "Unknown error" in the dialog (POST `/api/admin/schedule-publish`). Root cause: the two-state-schedule code (writes `divisions.schedule_visibility = 'published'`) was deployed to prod, but **mig 129 was never applied to prod** — the live prod CHECK still only allows `{unpublished, published_generic, published_teams}`, so the write hits a 23514 check_violation. Confirmed against live prod: constraint `age_groups_schedule_visibility_check` is still the 3-value version; only 1 division row exists (`unpublished`).
- **Why no self-serve fix:** There is NO platform-admin/ops surface that shows which migrations are applied per environment or warns "prod is behind dev." `check:migrations` is structurally blind to CHECK-clause-only migrations (mig 129 is exactly that), so the release gate passed even though prod was missing the schema change. A support person can't see, let alone close, this; it required a direct Management-API query against prod to diagnose. (This is the same class as the migration-040 incident.)
- **Suggested surface:** (1) A platform-admin/dev-tools "Migration drift" panel listing applied vs. expected migrations per env (dev/prod) with a red flag when prod is behind. (2) Harden `check:migrations` (or add a companion check) to detect CHECK-constraint-only migrations instead of silently skipping them, so the release gate can't green-light shipping CHECK-dependent code ahead of its migration.
- **Workaround used:** Diagnosed via a read-only Management-API query against prod; fix is to apply mig 129 to prod (`node scripts/apply-migration-api.mjs supabase/migrations/129_schedule_visibility_two_state.sql --prod`) then refresh snapshots — an engineering action.
- **Effort:** S (harden the migration check) / M (productized drift panel)
- **Related:** [[project_schedule_publish_simplify]] / [[reference_prod_supabase]] / release-gate (`scripts/check-prod-migration-drift.mjs` exists but isn't surfaced or enforced pre-deploy)

## Legacy playoff bracket can't be repaired without engineering — date TBD — ✅ ADDRESSED 2026-06-18
- **Resolution:** Dissolved by the Bracket Graph-Layout work (see [[project_bracket_graph_layout]]). Bracket columns + connectors now derive from the Winner/Loser feed graph, not the code string, so a legacy `G1..G7` (or any renamed) bracket renders as a correct tree for admins AND fans/PDF with **no data repair**. The `repair-legacy-bracket-codes.mjs` script is now optional (cleanup only). Remaining (lower-pri) gap: there's still no platform-admin UI to *inspect* a split `bracket_id` (advancement-breaking) — keep on the backlog as a small diagnostic surface.
- **Symptom:** A prod playoff bracket built by an older generator uses non-canonical `bracket_code` values (`G1..G7`). The renderer can't group those into round columns, so the bracket diagram shows each game in its own column in arbitrary order (e.g. G7 before G6) with crisscrossing connector lines — "links not working as expected." Customer wants to keep all game data (seeds, times, locations) but fix the wiring/display.
- **Why no self-serve fix:** There is NO platform-admin (or org-admin) surface to inspect or repair a bracket's underlying `bracket_code`/placeholder wiring. The inline Bracket Editor re-saves codes as-is (it preserves `bracket_code`), so it can't migrate a legacy scheme. A non-engineer has no path; this required a one-off Node repair script against prod.
- **Suggested surface:** A "Repair / re-code bracket" action in the Schedule playoff admin (or platform-admin org tools) that re-derives canonical `R{round}-{n}` codes from the Winner/Loser feed graph and rewrites placeholders in lockstep, preserving team-ids/dates/scores. Also surface a "distinct bracket_id" warning when one visual bracket is split across ids (advancement-breaking).
- **Workaround used:** `scripts/repair-legacy-bracket-codes.mjs` (dry-run by default, `--apply` to write) — graph-based re-code + lockstep placeholder rewrite, run against `.env.production.local`.
- **Effort:** M (data-repair tool) / L (productized admin action + the broader fix of making `bracketRoundInfo` order legacy codes by feed-graph depth)

**Scope:** Premium Coaches Portal (rep_* domain). Sport-neutral by construction (all vocabulary is coach-defined free text).

## Owner decisions (locked 2026-07-09)
1. **Ownership:** tag libraries + award lists are **per-team**, with an org-wide shared layer for League/Club standardization. **Revised 2026-07-13:** the shared layer is **org-authored** (admin curates one org-level library) — NOT bottom-up "promote a team tag." At 15+ teams, per-team promotion reintroduces the drift/duplication the feature exists to prevent and floods the admin with promotions to referee. See P3 for full rationale. (Underlying data shape is the same either way: `team_id NULL` = org-wide.)
2. **Permissions follow capabilities:** game tags require schedule access, money tags require money-write, awards require roster/schedule access; head coach always can. No head-coach-only creation lock.
3. **Awards are internal-only V1** (coach portal surfaces only). Public/parent-facing celebration deferred pending a consent-for-minors conversation.
4. **First shippable slice = game tags + the "record vs tag" report** (proves the tag → report loop end-to-end).

## Design principles
- **Two animals, not one:** *tags* are labels for grouping (games, expenses); *awards* are records (player + award + game/tournament + date). One shared library/curation pattern under the hood; never a generic "custom properties on everything" editor (that's the stacked-junk-drawer disease in data-model form).
- **Invisible until used:** entry points are contextual (a quiet tag affordance on the game/expense edit surface, a "give award" moment post-game). No new nav. Value surfaces in reports (Season Review / Insights hub, money reports).
- **Curation over creation:** rename + **merge** tooling ships in P1 (tag drift — "Top teams" vs "top in province" — is how these systems rot; merging must re-point history). Sane per-kind cap (~50) keeps libraries curated.
- **Insights-hub alignment:** game-tag analytics belong to the planned Insights hub (see COACH_LINEUPS_IA_AND_INSIGHTS_HUB_PLAN §Insights); this project supplies its first differentiated dataset, it does not build a new destination.

## Phases

### P1 — Game tags + "vs tag" report (first slice)
- **Data (migration 181, dev-only):**
  - `rep_team_tags` — id, org_id, team_id (nullable later for org-promoted), kind (`game` | `expense`), name (≤40 chars), created_by, timestamps. UNIQUE (team_id, kind, lower(name)). RLS mirrors `rep_team_lineup_templates` (mig 159): org members read, coaches on assigned teams + org admins write.
  - `rep_team_event_tags` — event_id FK → rep_team_events, tag_id FK → rep_team_tags, composite PK (no id/org_id/team_id — RLS reaches scope through tag_id, mirroring mig 071's `rep_team_lineup_entries` EXISTS pattern).
  - `merge_rep_team_tags(winner, loser)` — `SECURITY DEFINER` Postgres function (first atomic-merge precedent in the codebase, modeled on mig 169's `accept_tryout_and_create_dues`): re-points every event-tag link from loser→winner then deletes the loser, guarded same-team/same-kind. **[x] DONE.**
- **UI:** tag chips on the game edit/detail surface (schedule event form) with autocomplete-or-create (type to filter existing tags, Enter or click to toggle, "+ Create" when no match); a Tags manager modal (rename / merge / delete) reachable via a "Manage tags" link on the same surface. Capability: `capabilities.schedule` gates both applying and managing (no new capability). Read-only tag chips also show on the game detail slide-over. **[x] DONE.**
- **Report:** the "How are we doing?" Insights results report gains a tag filter-chip row (per tag: game count on the chip; selecting one filters the game table and shows W-L-T + RF/RA above it) with tap-through built in (the table row IS the filtered list). Self-hides per-chip at zero finalized games for that tag, and hides the whole row at zero team tags. Re-scoped from the original "own section" idea to fit Insights V3's tile-cap rule (new metric → existing report, not a new tile). **[x] DONE** (`app/[orgSlug]/coaches/teams/[teamId]/history/results/page.tsx`, 2026-07-10, uncommitted — owner approved an interactive mockup first).
- **API:** team-scoped tag CRUD (`/tags`, `/tags/[tagId]`) + merge endpoint (`/tags/merge`); events GET returns the team's tag library + an event→tags map; events POST/PATCH accept an optional `tagIds` array (validated against the team's library, replace-on-save). **[x] DONE** — the results report above is the first consumer of the events GET's `tags`/`tagsByEventId` fields.

### P2 — Player awards
- **Data (migration 182):**
  - `rep_team_award_types` — id, org_id, team_id, name (≤40 chars), emoji (short text, nullable — the icon-picker's chosen character or a custom-typed one), sort_order, is_active (bool, default true — retire = flip false, never a hard delete), created_by, timestamps. UNIQUE (team_id, lower(name)) WHERE is_active (a retired name can be reused). RLS mirrors `rep_team_tags` (mig 181): org members read, coaches on assigned teams + org admins write. Seed helper: on first read with zero rows for a team, seed MVP 🏆 / Best Hitter 💪 / Hustle Award 🔥 (editable starter list, not a hard-coded UI default — same "seed on first touch" convenience as elsewhere) rather than a migration-time backfill (a migration can't know future teams).
  - `rep_player_awards` — id, org_id, team_id, player_id FK → rep_roster_players, award_type_id FK → rep_team_award_types, event_id nullable FK → rep_team_events (the specific game, when tied to one), tournament_label nullable text (≤80 chars — free-text occasion when not tied to a single game; a row can have neither set, meaning a general/season recognition), awarded_at date, note nullable (≤200 chars), created_by, timestamps. No merge RPC needed (unlike tags) — award types are a small curated list a coach picks from, not free-typed per-game vocabulary, so name drift isn't the same risk; rename + retire is sufficient (owner-confirmed, not re-litigated).
  - Retiring an award type is a plain UPDATE (`is_active=false`), not a delete — every past `rep_player_awards` row keeps resolving the type's current name/emoji at render time (same as a rename does), so retiring or renaming never rewrites history, it just removes the type from the picker for new awards. "Restore" is the inverse UPDATE.
- **UI:**
  - **Give-award entry points (two):** (1) primary — inside the schedule page's game detail slide-over, directly below a new "Awards given" section, visible once the game has a final score (gated on scored status — can't award a game that hasn't been played). (2) secondary — a "+ Give an award" button on the new awards report page (below), for anything not tied to a single game (a whole tournament, mid-season recognition, catching up on one they forgot). Both open the same picker: player select (roster, active only) + award-type chip row (existing active types, click to pick, or "+ New" to create one on the fly with the icon picker) + optional note (≤200) + a "tied to" control (a specific past final game, a free-text tournament/occasion, or left general). **Save closes the modal** (owner correction, 2026-07-12 live testing — the original design kept it open and reset the fields for rapid consecutive entry; the owner found that surprising and wanted a normal submit-and-close. Giving a second award for the same game is just a re-tap of "Give an award" against the now-updated "Awards given" list, which is confirmation enough).
  - **Schedule list:** a small trophy-and-count badge on any game row that has awards given, mirroring the tag-chip precedent already on this list.
  - **Player profile (`roster/[playerId]`):** new "Awards" section alongside the existing Attendance/Dues sections — "N awards this season" headline + a list (icon, award name, game/tournament + date, note on tap). Empty state matches the Attendance section's "nothing yet" copy convention. Player-scoped, so it's inherently season-scoped the same way the roster itself is (a new `player_id` is minted every rollover).
  - **New Insights hub tile — "Who's earning it?"** (5th tile, alongside the existing 4; owner-confirmed placement, see above). Opens a new report page: a leaderboard (players ranked by total awards, with a per-type breakdown chip row per player), an award-type filter-chip row (reuses the Lineups/Results filter-chip idiom; clicking a type filters both the leaderboard and the history table below and swaps the basis line to "vs {type}: N given," mirroring the "vs tag" swap pattern exactly), a full reverse-chronological history table of every award given, and the "+ Give an award" / "Manage award types" entry points.
  - **Manage award types modal** (mirrors `TagManagerModal`, no merge tab): active list with a single "Edit" action per row (inline form edits name AND icon together, via the icon picker — not two separate actions), "Retire"; a collapsed "Retired (N)" section with "Restore"; a "+ Add award type" row (name + icon picker); cap note "N of 30 used." **Reachable ONLY from the "Who's earning it?" report page** (owner correction, 2026-07-12 — an earlier pass also surfaced a "Manage award types" link on every game's slide-over; the owner pointed out award TYPES are a once-a-season setup task, unlike GIVING an award which happens constantly, so cluttering every game screen with a rarely-used management link was the wrong call. Removed from the schedule page entirely; the report page was already the right home and needed no change).
  - **Icon picker** — a small reusable component (not a new dependency/library): a fixed curated set of ~28 sport/achievement emoji (trophies, medals, gear, energy/attitude marks) rendered as a click-to-select grid, plus a "type your own" text fallback for anything outside the set. Used identically at award-type creation and at edit time.
  - Internal-only — no public/parent-facing surfaces anywhere in this phase (locked decision, unchanged).
- **API:** team-scoped award-type CRUD (`/award-types`, `/award-types/[typeId]` — PATCH covers rename/icon/retire/restore, no separate endpoints per action) + award records (`/awards` GET list + POST give, `/awards/[awardId]` DELETE to undo a mis-click); events GET gains a lightweight award-count-by-event map (mirrors `tagsByEventId`) for the schedule list's trophy badge; player detail GET gains an awards summary (mirrors the existing attendance/dues summary shape) for the profile section.
- **Capability:** `capabilities.schedule || capabilities.roster !== 'off'` gates give/manage/view — any coach who can already see the schedule or the roster can do this, matching the "no special locks" decision. Head coach always qualifies.
- **Caps:** 30 award types per team (soft, in-process check — same TOCTOU-acceptable convention as tags/lineup templates).

### P3 — Money tags + org-authored shared library — BUILT 2026-07-13 (dev, uncommitted; mig 184 dev-only)
**Verification:** `npm run typecheck` clean; focused lint clean (0 new warnings beyond each file's pre-existing patterns); `check:dictionary` passes (new `rep_team_expense_tags` + nullable `team_id` documented); snapshots refreshed to watermark #184. Additive, necessary extension of the P2 award endpoints so org-shared award types are recognized (award-types GET, awards GET/POST now use `getRepTeamAwardTypeLibrary`/pass orgId) — flagged to owner, not a silent P2 behavior change. **Next: owner browser test → `/review` → `/docs` → commit decision. ⚠ mig 184 PROD-PENDING.**

- **Money tags (kind `expense`):** reuse the P1 `rep_team_tags`/`merge` machinery with `kind='expense'`; a NEW `rep_team_expense_tags` join table (the P1 `rep_team_event_tags` is FK'd specifically to `rep_team_events`, so expenses need their own junction). Applied on the Add Expense + Add Payable forms and re-taggable inline on the expense list; expense list + Budget vs. Actual gain a tag filter-chip row (mirrors the P1 "vs tag" idiom exactly). **Gated on the MONEY capability** (`canViewMoney`/`canWriteMoney`), not schedule — this is the one deliberate difference from game tags. (Note: the "subcategory" ask is largely served today by the existing Category → Item taxonomy incl. coach-created custom items — money tags are the lowest-marginal-value slice, hence last.)
- **Tag picker upgrade:** money tags use a new reusable **ADO-style search-combobox** (a search box + type-ahead dropdown of the already-loaded library, only *selected* tags shown as chips) rather than P1's filtered-chip-list picker — money-tag libraries grow much larger than game tags, so a wall of pills doesn't scale. Zero extra API calls (library loads once with the page; filtering is client-side). The committed P1 game-tag picker is left untouched this phase (could adopt the combobox later).
- **Org-authored shared library (REPLACES the originally-scoped bottom-up "promote a team tag" model — owner decision 2026-07-13).** Rationale: at 15+ teams, letting every team promote its own tags org-wide reintroduces exactly the drift/duplication the whole feature fights, and buries the admin in promotions to referee across N team lists. Instead the org **owner/admin authors** a small shared library (game tags, money tags, award types) from ONE org-level screen (Admin → Rep Teams → Shared library). Shared items are stored with `team_id NULL, org_id` set and surface in every team's picker; team-private tags sit side-by-side, unchanged. Coaches apply shared items but cannot rename/delete them (RLS: coach writes require `team_id` ∈ their teams, which NULL fails; org admins manage them). **No bottom-up promote path** — cleaner, scales, stays true to curation-over-drift.
  - **Signal = colour, not a badge** (owner refinement 2026-07-13): org items render blue, team items green, with a one-line legend where both appear (dropdown squares → selected chips → row chips → filter chips), rather than an "ORG" stamp on every pill.
  - **Schema (migration 184, dev-only):** widen `rep_team_tags.team_id` and `rep_team_award_types.team_id` to NULLABLE; add partial unique indexes for the org-wide rows (`(org_id, kind, lower(name)) WHERE team_id IS NULL` for tags; `(org_id, lower(name)) WHERE is_active AND team_id IS NULL` for award types). Existing org-admin RLS write policies already cover the `team_id NULL` rows via the `org_id`-membership check — no new policies needed. `getRepTeamTags`/`getRepTeamAwardTypes` widen to return team + org-wide (`team_id = $team OR (team_id IS NULL AND org_id = $org)`).
  - Org-level cross-team rollups become possible later (out of scope here).
